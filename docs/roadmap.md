# LifeOS 发展路线图 (Roadmap)

本文档定义了 LifeOS 从工具化向“魔女模拟器”游戏化演进的 5 个关键阶段。

## Phase 1: 工具与数据中枢 (Current)

**目标**：建立稳固的底层架构与核心生活管理工具。

- **交付物**：
  - **Tab 导航**：Expo Router 底部 Tab + 模块内 Stack。
  - **生活工具集**：具备基础功能的习惯打卡、待办。
  - **Zustand 中央存储**：实现跨模块共享的 `habitStore`、`todoStore`、`gameStore`（占位）。
  - **本地持久化**：使用持久化方案确保数据在应用重启后不丢失。
  - **工程底座**：明确的模块引用规范与目录边界，防止代码耦合。

## Phase 2: 游戏引擎 MVP（最小剧情链路）

**目标**：一张小地图 + 5–10 个事件 + 分支选项 + 属性变化 + 存档。

- **交付物**：
  - **数据驱动事件格式**：将剧情/事件/地图组织为内容包。
  - **引擎解释器**：条件判断、效果执行、事件/地点跳转。
  - **基础 UI**：事件文本框与选项列表。
  - **GameStore 存档**：提供 save/load 接口与基础存档槽。

## Phase 3: 内容生产效率升级

**目标**：新增剧情不改引擎代码；提供校验/预览能力。

- **交付物**：
  - **内容校验工具**：事件 id 查重、跳转目标存在性检查、条件/效果字段合法性校验。
  - **预览/编辑体验**：简单的内容校验脚本或工具页，支持快速预览分支。

## Phase 4: 扩展模块与存储升级

**目标**：支持碎片记录与搜索；随数据增多平滑升级存储。

- **交付物**：
  - **Fragments 模块**：灵感/心情碎片记录与按关键词筛选。
  - **存储升级**：从 `AsyncStorage` 平滑迁移到 SQLite/全文检索（按需）。

## Phase 5: AI（先占位，后接真实 API）

**目标**：保持 AI 架构可插拔、可替换；当前不接入真实 API。

- **交付物**：
  - **`aiClient`** **统一接口** 与 provider 占位/Mock（Gemini/DeepSeek 等）。
  - **角色卡与 prompts 规范**：定义存放与拼装规则。
  - **“当日状态摘要”接口**：从中央数据库汇总可供 AI 使用的简短摘要。
