## 1) 总体设计目标（给小白的“护栏”）

- **模块要“像插件”**：工具、游戏、AI 互不缠绕，但能通过同一套数据层互通。
- **数据是“中央数据库”**：所有模块只通过统一的数据读写接口共享信息（AI 读取习惯、日记搜索、游戏存档等），避免组件之间互相传参到失控。
- **强约束目录 + 强约束边界**：任何“跨模块引用”都必须走公共层（`core/`、`stores/`、`services/`），不允许 A 模块直接 import B 模块内部文件。

## 2) 技术选型（与你的选择对齐）

- **路由/导航**：Expo Router（文件即路由）。
  - 大模块：底部 Tab（工具 / 游戏 / AI / 扩展 / 设置）。
  - 模块内：各自 Stack（例如工具里有 打卡/待办/详情页）。
- **状态管理（中央数据库）**：推荐 **Zustand + 持久化** 作为“全局数据总线”。
  - 原因：学习成本低、写法直观、跨模块共享自然；后期可逐步引入更严格的分层（selectors、slices）。
  - Context 适合少量 UI 状态，不适合承载你的“全局数据互通”。
- **本地持久化**：优先走本地（离线优先）。
  - Phase 1/2 可用 `AsyncStorage` 级别持久化；当数据变多（聊天记录、事件库、日记）再升级到 SQLite/WatermelonDB/Realm（在路线图里分阶段升级）。

## 3) 信息架构（IA）与导航草图

- **底部 Tab（一级模块）**
  - **工具**：习惯打卡、待办（未来扩展更多记录工具）
  - **游戏**：魔女模拟器（地图/事件/养成/存档）
  - **AI**：角色聊天（先做架构占位；后续可读取“今日打卡”等全局数据进行反馈）
  - **扩展**：灵感碎片/心情碎片/搜索筛选
  - **设置**：数据导入导出、AI Key 管理（未来）、主题等
- **Expo Router 路由约定**（概念）
  - `(tabs)/tools/`*
  - `(tabs)/game/`*
  - `(tabs)/ai/`*
  - `(tabs)/lab/`*（扩展/实验室）
  - `(tabs)/settings/`*

## 4) “中央数据库”怎么让数据互通（关键）

把所有“跨模块会用到的东西”抽成**领域数据（Domain Data）**，用 store 统一管理：

- **HabitDomain**：习惯定义、每日打卡记录、连续天数等
- **TodoDomain**：待办、标签、完成记录
- **Journal/FragmentsDomain**：灵感碎片、心情碎片、全文检索索引（后期）
- **AIChatDomain**：角色卡、人设、聊天会话列表、消息、摘要
- **GameDomain**：玩家属性、背包、地图位置、事件触发状态、存档槽

统一约束：

- UI（screens/components）只能 **读写 store 提供的 actions**，不直接操作存储。
- 持久化/迁移逻辑在 `persistence/` 或 `services/storage/`，store 不关心底层是 AsyncStorage 还是 SQLite。

## 5) 游戏引擎底座（数据驱动，像写小说一样加剧情）

目标：你未来新增剧情时，**尽量只写配置/文本，不改引擎代码**。

### 5.1 核心思路

- **引擎代码只做“解释器/执行器”**：读取事件配置 → 根据条件判断 → 展示文本/图片/选项 → 应用效果（属性变更、物品增减、地图移动、触发标记）。
- **内容数据独立成“内容包”**：每章/每区域/每角色一套文件夹，便于管理、版本化、替换。

### 5.2 建议的数据模型（概念层）

- **World/Map**：地图节点（地点）+ 连边（可走路径）
- **Event**：事件节点（文本、图片、选项、条件、效果）
- **Condition**：触发条件（属性阈值、flag、时间、地点、物品等）
- **Effect**：效果（改属性、给道具、设置 flag、跳转事件/地点、推进时间）
- **PlayerState**：玩家属性（数值/标签/状态）与库存

### 5.3 内容组织方式（让你“像写小说”）

- 每个事件一个文件（或一个章节一个文件，视规模而定）：
  - 事件 id、标题、正文段落、可选项、每个选项的条件与效果
- 图片资源与事件同目录，保持相对路径一致
- 用“flag”记录分支与一次性事件

### 5.4 编辑体验（后续可升级）

- Phase 2 先用 JSON/YAML + 你手写文本即可跑通。
- Phase 3/4 可做一个“内容编辑器”（哪怕很简陋）：表单化编辑事件、校验 id、预览分支。

## 6) AI 模块预留（Gemini/DeepSeek 可替换）

目标：AI 是可插拔的 provider，不把业务绑死在某一家 API。

### 6.1 分层

- **ai/providers/**：不同厂商的实现（GeminiProvider / DeepSeekProvider）
- **ai/aiClient.ts**：统一接口（`sendMessage`、`streamMessage`、`listModels` 等概念）
- **ai/prompts/**：系统提示词、人设模板、工具调用提示（后期）
- **ai/memory/**：聊天上下文管理（窗口裁剪、摘要、长期记忆）

### 6.2 人设与上下文怎么管

- **角色卡（Character Card）**：独立数据（名称、背景、口吻、禁忌、关系、示例对话）。
- **System Prompt**：由“角色卡 + App 规则 + 当日状态摘要”拼装生成。
- **当日状态摘要**：从中央数据库读取（例如：今日习惯完成度、连续天数、待办完成数），做成一段短摘要喂给 AI（不是直接把全量数据塞进去）。
- **上下文策略**：
  - 短期：最近 N 轮对话
  - 长期：定期摘要（summary）+ 关键事实（facts）

## 7) 文件夹结构清单（防屎山版本，按职责切）

下面是一套**可长期扩展**且适配 Expo Router 的结构；核心原则是：

- `app/` 只放“页面路由壳”，不放业务。
- 业务按“领域模块”放 `features/`，共享能力放 `core/`。
- `app/`
  - `(tabs)/`：一级 Tab 路由组
    - `tools/`：工具相关页面入口（打卡、待办等）
    - `game/`：游戏页面入口
    - `ai/`：AI 聊天页面入口
    - `lab/`：扩展/实验室入口（灵感碎片、搜索等）
    - `settings/`：设置页
  - `_layout.`*：全局导航壳（tab/stack 组合）
- `src/`
  - `core/`：跨模块公共层（最重要）
    - `types/`：全局类型
    - `constants/`：常量、枚举、配置
    - `utils/`：纯工具函数
    - `ui/`：跨模块通用 UI（按钮、弹窗、表单控件）
    - `theme/`：主题、颜色、字体
  - `stores/`：中央数据库（Zustand slices）
    - `habitStore.`*
    - `todoStore.`*
    - `aiStore.`*
    - `gameStore.`*
    - `fragmentsStore.`*
    - `index.`*（组合/导出）
  - `services/`：对外部/底层能力的封装
    - `storage/`：本地存储适配层（后期可从 AsyncStorage 升级到 SQLite）
    - `ai/`：AI 统一 client + providers
    - `analytics/`：埋点（可选，后期）
  - `features/`：按领域分模块（业务都在这）
    - `tools/`
      - `habits/`：打卡业务（计算连续天数、完成率）
      - `todos/`：待办业务
    - `game/`
      - `engine/`：引擎（解释器、条件判断、效果执行、存档）
      - `content/`：剧情/事件/地图数据（纯数据文件）
      - `ui/`：游戏专用 UI（事件框、选项卡、地图组件）
    - `ai/`
      - `characters/`：角色卡数据与管理
      - `chat/`：聊天会话/消息 UI 与业务
      - `prompting/`：system prompt 组装
    - `fragments/`
      - `capture/`：碎片记录
      - `search/`：关键词筛选/检索（后期）
  - `assets/`：图片、字体、音效（可与 `features/game/content` 协调组织）
- `docs/`
  - `architecture.md`：你的“白皮书”落地文档（你随时可回来看）
  - `game-content-guide.md`：写剧情的规范（命名、id、分支约定）

## 8) 分阶段开发路线图（4–5 个 Phase，稳妥搭积木）

### Phase 1：先把“工具+数据中枢”跑起来（最快见成果）

- 目标：你能每天打卡、写待办；数据能保存；界面有基本框架。
- 交付物：Tab 导航、Habit/Todo 两个最小功能、Zustand store、基础持久化。

### Phase 2：游戏引擎 MVP（最小剧情链路）

- 目标：一张小地图 + 5–10 个事件 + 分支选项 + 属性变化 + 存档。
- 交付物：数据驱动事件格式、引擎解释器、基础 UI（事件框/选项）、GameStore 存档。

### Phase 3：内容生产效率升级（让你“像写小说”）

- 目标：新增剧情不改引擎代码；有校验/预览能力。
- 交付物：内容校验脚本/工具页（在 app 内或开发脚本）、事件 id 查重、条件/效果的错误提示。

### Phase 4：扩展模块（灵感碎片 + 搜索）与存储升级

- 目标：碎片记录、按关键词筛选；数据变多后升级到 SQLite/全文检索（如需要）。
- 交付物：Fragments 功能、搜索体验、存储层从 AsyncStorage 平滑迁移。

### Phase 5：AI（先占位，后接真实 API）

- 目标：保持 AI 架构可插拔、可替换；当前不接入真实 Gemini/DeepSeek API，仅保留目录与统一接口壳，避免未来返工。
- 交付物：`aiClient` 统一接口、provider 占位（空实现/Mock）、角色卡与 prompts 的存放规范、从中央数据库生成“当日状态摘要”的数据接口（供未来接入时直接使用）。

## 9) 关键“防屎山”规则（你可以当项目宪法）

- 页面只负责展示与交互，业务逻辑必须在 `features/` 或 `stores/`。
- 跨模块共享只通过 `stores/` 与 `services/`，不允许互相 import 内部文件。
- 游戏内容必须是数据文件（章节/地点/角色拆包），引擎代码禁止写死剧情文本。
- AI provider 可替换：业务永远只调用 `aiClient` 的统一接口。

## 10) UI 布局与交互全案（Builder 执行依据）

### 10.1 指导原则（骨架先行）

- 本阶段目标是把页面结构、信息层级、核心交互与导航流统一起来，让后续“填细节/加功能/打磨视觉”都在同一套骨架上迭代。
- 页面只做展示与交互，业务计算与状态读写必须走 `stores/` 与 `features/`；任何未来 AI 联动只预留接口，不在 UI 层写死规则与文案拼装逻辑。
- 所有模块统一遵循：打开即用（高频入口减少层级）、主按钮明确（每页只允许 1 个“主按钮”）、反馈克制（轻提示优先，强弹窗只用于风险操作）。

### 10.2 全局风格（气质与视觉系统）

- 整体气质：温柔治愈 + 轻魔女感 + 少量轻毒舌；文案表现为“陪伴式提醒 + 偶尔吐槽”，避免嘲讽与负向攻击。
- 强调色：以 `#D1BBDE` 柔和紫作为全局主强调色；允许使用蓝色/黄色作为少量点缀色，但不得抢占主按钮与关键状态的识别权。
- 页面结构语言：统一采用“卡片化分区 + 清晰标题 + 轻分割”的信息组织方式；高频操作按钮保持在拇指可达区域。
- 模块气质分工：
  - 工具：清晰高效（任务与动作直达）
  - 游戏：书卷气 / 地图感 / 沉浸叙事（更像打开一本书或地图）
  - AI：陪伴感（后续细化）
  - 扩展：轻量记录（不打扰主流程）
  - 设置：朴素清楚（可理解、可恢复）

### 10.3 页面架构（信息层级与导航规则）

- 全局导航采用 Tab 作为一级模块入口：工具 / 游戏 / AI / 扩展 / 设置；模块内部使用 Stack 管理页面流。
- 每个页面统一包含三段式信息层级：
  - 顶部区：页面标题 + 一句解释（可选）
  - 主内容区：2–4 个卡片/分区，按照“最重要→次要→辅助”的顺序排列
  - 底部主操作区：主按钮与高频快捷入口（如果有）
- 导航跳转规则：
  - 总览页只负责“概览 + 跳转”，不承载复杂编辑。
  - 详情/清单页负责“快速编辑 + 快速完成 + 快速新增”。
  - 游戏流只允许“继续/进入地图/进入事件”三类主入口，避免混入工具型操作。

### 10.4 工具 Tab（总览页 + 高速操作页）

#### 10.4.1 工具总览页（首页）

- 定位：高频入口页，打开即用；只展示关键摘要与最短操作链路。
- 页面分区（三块固定）：
  - Hello World 区：
    - 显示日期（人类可读）、天气占位（后接 API）、角色一句话（可带轻毒舌口吻）
    - 示例文案：“让我歇会吧~”
  - 魔女能量汇总区（习惯总览）：
    - 用圆环或精致进度条展示今日习惯完成百分比与 `done/total`
    - 点击卡片主体进入“习惯页”
    - 提供“快速新增习惯”入口（不进入多层编辑）
  - 魔女事务处理区（待办总览）：
    - 展示今日最重要的 3–5 个待办（按创建顺序或未完成优先）
    - 支持快速完成（在列表内一键切换 done）
    - 点击卡片主体进入“待办页”
    - 提供“快速新增待办”入口
- 语境通知官预留：工具总览页顶部或 Hello 区下方预留“小弹窗/小横幅”显示位；本阶段只占位，不接真实 AI。

#### 10.4.2 习惯页（清单页）

- 定位：习惯的高频清单 + 当日打卡；减少层级，突出“今天完成了什么”。
- 信息分区：
  - 顶部：标题 + 今日完成统计（done/total）
  - 主内容：习惯列表（每条包含标题、可选 schedule、连续天数、完成按钮）
  - 主操作：快速新增（弹出轻量表单）；编辑仅在必要字段内进行（标题/计划/归档）
- 交互规则：
  - 单击完成按钮：切换今日 done（轻提示即可，不弹窗）
  - 新增/编辑：使用轻量弹层（底部 sheet）承载输入
  - 归档：允许弹窗确认（避免误操作）

#### 10.4.3 待办页（清单页）

- 定位：今日事务处理；强调快速完成与快速新增。
- 信息分区：
  - 顶部：标题 + 今日完成统计（done/total）
  - 主内容：待办列表（显示标题、due、tags；点击条目快速完成）
  - 主操作：快速新增（新增后允许在列表内再补充 due/tags）
- 交互规则：
  - 点击条目：快速完成/取消完成（轻提示）
  - 删除：强弹窗确认（风险操作）

### 10.5 游戏 Tab（书卷气入口 + 地点跳转式地图）

#### 10.5.1 游戏首页

- 主按钮：以“继续游戏”为唯一主按钮；若无存档则展示“开始新旅程”为主按钮（语义仍属于继续/开始）。
- 视觉方向：拒绝工具型列表；采用羊皮纸纹理或柔和紫渐变作为背景倾向；中间放“可交互地图节点插画占位”作为世界入口。
- 核心入口：
  - 继续游戏 → 进入事件页（继续当前事件）
  - 世界入口（地图节点）→ 进入地图页

#### 10.5.2 地图页（初版）

- 交互形式：地点按钮/图片按钮列表；点击地点即“跳转到地点 → 进入该地点相关事件”。
- 限制：不做自由探索、拖拽大地图、迷雾与随机探索；仅提供“地点跳转式地图交互”。

#### 10.5.3 事件页（MVP 执行链路）

- 职责：文本事件展示、可选项过滤（条件）、效果应用、跳转；并提供最低限度存档入口。
- 存档：提供保存/读取入口即可，不做复杂 UI 与多槽管理扩展。

### 10.6 AI Tab（暂不细化）

- 本阶段仅保留架构占位与后续接入空间；不对首页 UI 做深度设计与开发。

### 10.7 扩展 Tab（结构预留）

- 灵感碎片与心情碎片必须分开为两条页面流，不合并。
- 本阶段只做入口与页面骨架，暂不做复杂搜索体验与全文检索 UI。

### 10.8 Contextual Messenger（语境通知官）预留

- 形态：非聊天页、非常驻；以小弹窗/小横幅在特定页面的合适时机出现。
- 本阶段要求：只做 UI 预留与触发入口，不接真实 AI。
- 后续联动方向示例：
  - 数据联动：连续 3 天未打卡时，进入工具页自动弹出：“再不喝水，我就要把你的法杖变成咸鱼了。”
  - 环境联动：接入天气 API 后，雨天触发关于潮湿、忧郁等主题的角色化提醒。

### 10.9 反馈与弹窗边界（统一交互语法）

- 轻提示（不打断）适用：快速完成、快速新增成功、轻量状态切换（done/undone、简单跳转）。
- 小弹窗/小横幅适用：语境通知官、系统提示类反馈（非风险）。
- 强提醒弹窗（打断）仅适用：破坏性操作（删除/归档/清空）、可能导致不可逆数据变更的操作（覆盖存档槽）。

### 10.10 Builder 当前 UI 落地优先级（本阶段验收口径）

- 优先细化并落地：工具总览页、习惯页、待办页、游戏首页、游戏地图页初版。
- 暂不细化：AI 首页。
- 扩展模块：先保留灵感碎片页与心情碎片页分离结构；不急着做复杂搜索体验。
- 验收只看：页面骨架、信息分区、导航交互可用；视觉细节后续再迭代。
